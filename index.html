Q1 Define the role of workspace management in Jenkins. 2 Marks
Dedicated Build Directory: A Jenkins workspace is a dedicated directory on the build agent where Jenkins checks out source code and executes build steps for each job. It isolates each project's files to avoid conflicts between concurrent builds.
Maintenance & Cleanup: Workspace management prevents disk space issues by cleaning up stale artifacts. Jenkins supports manual cleanup via the UI, the Workspace Cleanup Plugin for automatic removal, and post-build actions to delete workspace files after job execution — ensuring reliable, reproducible builds every time.

Q2 Identify the significance of Group ID and Artifact ID within a Maven project. 2 Marks
Group ID: Uniquely identifies the organization or project group (e.g., com.example). It follows the reverse-domain naming convention and groups related projects under one namespace, enabling Maven to locate the correct repository directory for dependency resolution.
Artifact ID: Identifies the specific project or module (e.g., my-app). Combined with Group ID and version, it forms a globally unique coordinate (groupId:artifactId:version) used by Maven to download, cache, and resolve dependencies from local and remote repositories.
<groupId>com.example</groupId> <artifactId>my-app</artifactId> <version>1.0.0</version>
Q3 Outline the primary function of Docker Hub in container management. 2 Marks
Central Image Registry: Docker Hub is a cloud-based registry that acts as the default storage and distribution platform for Docker images. It hosts official images (Ubuntu, Nginx, MySQL), community images, and private repositories — allowing developers to docker pull ready-to-use images or docker push custom images for sharing.
CI/CD Integration & Collaboration: Docker Hub supports automated builds triggered from GitHub/Bitbucket, image versioning via tags (e.g., nginx:1.21), and webhook integration. Teams can collaborate using shared private repositories with role-based access control, making it a critical hub in the DevOps pipeline.
Q4 What is the necessity of maintaining a secure deployment pipeline? 2 Marks
Preventing Unauthorized Access & Vulnerabilities: A secure deployment pipeline enforces authentication, role-based access control, and encrypted communications (HTTPS/TLS) at every stage — from code commit to production deployment. This prevents unauthorized code injection, supply chain attacks, and exposure of sensitive credentials in the pipeline.
Ensuring Compliance & Reliability: Security gates (automated SAST/DAST scans, image vulnerability scanning, signed artifacts) embedded in the pipeline ensure only validated, compliant code reaches production. This maintains software integrity, meets regulatory requirements, and reduces the risk of deploying broken or malicious builds.
Q5 Summarize the prerequisites for configuring a Maven build environment. 2 Marks
Java Development Kit (JDK 8+): Maven requires JDK 8 or later to be installed. The JAVA_HOME environment variable must be set to the JDK installation path. Verified using java -version.
Environment Variables & Network: M2_HOME must point to the Maven installation directory, and Maven's bin/ folder must be added to the system PATH. An internet connection (or configured proxy in settings.xml) is required to download dependencies from Maven Central Repository.

Q6 Illustrate the execution flow of parent and child builds within a Jenkins Build Pipeline. 5 Marks
Parent–Child Build Execution Flow
Parent Job Definition: The parent build is the primary Jenkins job. It handles initial steps such as SCM checkout, compilation, and code quality checks. After its own steps complete, it triggers downstream (child) jobs via Post-Build Actions → Build Other Projects.
Child Job Triggering: Child jobs are dependent sub-jobs (e.g., Unit Test Job, Integration Test Job). They are configured as downstream builds and can be triggered based on the parent's build result (always, on success, on failure).
Parallel Execution: Unlike sequential builds, parent-child builds allow child jobs to run in parallel, significantly reducing total pipeline time. This is ideal for modular projects where independent components can be tested simultaneously.
Parameterized Trigger: Parent jobs can pass parameters (like build version, environment name) to child jobs using the Parameterized Trigger Plugin, enabling flexible, context-aware child build execution.
Pipeline Script Example:
pipeline { agent any stages { stage('Parent Build') { steps { echo 'Building Parent Job' build job: 'Child_Job_A' build job: 'Child_Job_B' } } } }
Best Practice: Use Declarative Pipelines with parallel blocks for child jobs, store Jenkinsfiles in version control, and configure webhook-based triggers for automated execution on each commit.
Q7 Explain dependency management principles used to configure and manage external libraries in Apache Maven for a Java project, with suitable examples. 5 Marks
Maven Dependency Resolution Flow
Declaring Dependencies in pom.xml: All external libraries are declared inside the <dependencies> block using groupId, artifactId, and version. Maven automatically downloads and caches them.
<dependencies> <dependency> <groupId>org.springframework</groupId> <artifactId>spring-core</artifactId> <version>5.3.10</version> </dependency> </dependencies>
Dependency Scopes: Maven supports scopes to control dependency availability — compile (default, available everywhere), test (only during testing, e.g., JUnit), provided (provided by JDK/server, e.g., Servlet API), and runtime (needed at runtime, not compile time).
Transitive Dependency Resolution: Maven automatically resolves transitive dependencies — if Library A depends on Library B, Maven includes B without manual declaration. This prevents dependency conflicts and version mismatches.
Dependency Management Block: In multi-module projects, the <dependencyManagement> section in the parent POM centralizes version definitions, ensuring all child modules use consistent versions without individually specifying them.
Repository Configuration: Maven resolves dependencies from the Local Repository (~/.m2/repository) first, then Maven Central, then configured remote/enterprise repositories (Nexus/Artifactory). Custom repositories are added via <repositories> in pom.xml or settings.xml.
Snapshot vs Release: Dependencies versioned as 1.0-SNAPSHOT are development versions that Maven re-downloads on every build to pick up latest changes, while release versions are fixed and cached permanently — enabling controlled dependency lifecycle management.
Q8 Demonstrate the procedure for creating a new Docker container using a base image. 5 Marks
Docker Container Creation Process
Step 1 – Pull a Base Image from Docker Hub: Begin by pulling a verified base image. This forms the foundation layer of your container.
docker pull ubuntu:latest
Step 2 – Write a Dockerfile: A Dockerfile defines instructions to build a custom image on top of the base image — installing packages, copying files, setting environment variables, and specifying the startup command.
FROM ubuntu:latest RUN apt-get update && apt-get install -y python3 COPY app.py /app/app.py WORKDIR /app CMD ["python3", "app.py"]
Step 3 – Build the Docker Image: Use docker build to create the image from the Dockerfile. The -t flag assigns a name/tag.
docker build -t my-python-app:1.0 .
Step 4 – Run a Container from the Image: Launch a container instance. Use -d for detached mode, -p for port mapping, and --name for a custom container name.
docker run -d -p 5000:5000 --name myapp my-python-app:1.0
Step 5 – Verify & Manage the Container: Check running containers, view logs, inspect details, and stop/remove when done.
docker ps # list running containers docker logs myapp # view container output docker stop myapp # stop container docker rm myapp # remove container
Key Concept – Layered Architecture: Docker uses a layered filesystem. Each instruction in the Dockerfile creates a new immutable layer. The container adds a thin writable layer on top of the read-only image layers, enabling efficient storage and reuse.
Q9 Illustrate the node clustering capabilities provided by Docker Swarm. 5 Marks
Docker Swarm Cluster Architecture
Cluster Initialization: Docker Swarm groups multiple Docker hosts into a single logical cluster. One node acts as the Manager; others are Workers. Initialized with: docker swarm init --advertise-addr <MANAGER_IP>. Worker nodes join using a token: docker swarm join --token <TOKEN> <MANAGER_IP>:2377.
Manager Node Role: The Manager Node handles orchestration — scheduling tasks, maintaining cluster state in a Raft-distributed database, and managing service definitions. At least 3 Manager nodes are recommended for high availability (fault tolerance via Raft consensus).
Worker Node Role: Worker nodes execute tasks (containers) assigned by the Manager. They do not participate in scheduling decisions. Workers can be added/removed dynamically: docker node ls lists all nodes in the cluster.
Service Deployment & Scaling: Applications are deployed as services with defined replica counts. Swarm distributes replicas across worker nodes for load balancing: docker service create --name web --replicas 3 -p 80:80 nginx. Scaling is done with docker service scale web=5.
Fault Tolerance & Self-Healing: If a worker node or container fails, Swarm automatically reschedules the tasks onto healthy nodes to maintain the desired replica count — providing built-in self-healing and high availability.
Overlay Networking & Load Balancing: Swarm creates an internal routing mesh and overlay network, enabling containers across different nodes to communicate securely. Incoming traffic is automatically distributed among replicas using built-in load balancing — no external load balancer needed for basic deployments.
Q10 Demonstrate the necessary commands to interact with a running Docker container's command-line interface. 5 Marks
Launch Interactive Container: Start a container with an interactive terminal directly using the -it flag (interactive + TTY). This drops you into the container's shell.
docker run -it ubuntu bash # -i → keep STDIN open -t → allocate a TTY terminal
Execute Commands in a Running Container: Use docker exec to run commands inside an already-running container without stopping it. This is the primary method to access a running container's CLI.
docker exec -it <container_id> bash # Example: docker exec -it myapp /bin/bash
Attach to a Running Container: docker attach connects your terminal to a running container's primary process (PID 1). Unlike exec, attaching to the main process means exiting the shell may stop the container.
docker attach <container_id>
View Container Logs & Processes: Inspect the container's stdout/stderr output and monitor running processes without entering the container shell.
docker logs -f <container_id> # follow live logs docker top <container_id> # show running processes docker stats <container_id> # live CPU/memory stats
Copy Files To/From Container: Transfer files between the host filesystem and a running container's filesystem without needing a shell session.
docker cp myapp:/app/output.log ./output.log # container → host docker cp ./config.json myapp:/app/config.json # host → container
Inspect & Debug Container: Get detailed JSON-formatted metadata about a container — IP address, mounts, environment variables, network settings — useful for debugging configuration issues.
docker inspect <container_id> docker inspect --format='{{.NetworkSettings.IPAddress}}' myapp
Exec vs Attach – Key Difference: docker exec starts a new process inside the container (preferred for interactive debugging); docker attach connects to the container's main process (PID 1). Use exec to avoid accidentally killing the container on exit.

Q11 Explain the sequential phases of the Maven Build Life Cycle and POM configurations in automating a software build process. 10 Marks
Maven Build Lifecycle – Three Built-in Lifecycles
Overview of Maven Lifecycles: Maven has three built-in lifecycles — Clean (removes previous build artifacts), Default (main build + deployment process), and Site (generates project documentation). Each lifecycle is an ordered sequence of phases; invoking a phase automatically triggers all preceding phases.
Default Lifecycle – Phase 1: validate: Verifies that the project is correct and all necessary information is available (pom.xml is valid, dependencies are declared). If validation fails, the build stops immediately. Command: mvn validate.
Default Lifecycle – Phase 2: compile: Compiles the project's Java source files (src/main/java) into bytecode, placing .class files in the target/classes directory. Uses the maven-compiler-plugin. Command: mvn compile.
Default Lifecycle – Phase 3: test: Executes unit tests from src/test/java using frameworks like JUnit or TestNG via the maven-surefire-plugin. Build fails if tests fail. Tests can be skipped with -DskipTests. Command: mvn test.
Default Lifecycle – Phase 4: package: Packages the compiled code into a distributable format — .jar (Java Archive) or .war (Web Archive) — and places it in the target/ directory. The packaging type is defined in pom.xml with <packaging>jar</packaging>. Command: mvn package.
Default Lifecycle – Phases 5–7: verify, install, deploy: verify runs integration tests and checks package validity. install installs the artifact into the local ~/.m2/repository for use by other local projects. deploy copies the final artifact to a remote repository (Nexus/Artifactory) for sharing with teams. Command: mvn install.
POM (Project Object Model) – Core Configuration: The pom.xml is Maven's heart. It defines project coordinates (groupId, artifactId, version), packaging type, dependencies, plugins, and build configurations. All lifecycle behavior is controlled through pom.xml.
<project> <modelVersion>4.0.0</modelVersion> <groupId>com.example</groupId> <artifactId>my-app</artifactId> <version>1.0.0</version> <packaging>jar</packaging> <dependencies> <dependency> <groupId>junit</groupId> <artifactId>junit</artifactId> <version>4.13</version> <scope>test</scope> </dependency> </dependencies> </project>
Plugin Configuration in POM: Maven plugins extend lifecycle behavior. For example, maven-compiler-plugin sets the Java source/target version, maven-surefire-plugin controls test execution, and maven-jar-plugin configures the main class for executable JARs.
Clean Lifecycle: Executed with mvn clean, it deletes the target/ directory containing all previous build artifacts — ensuring a fresh, reproducible build. Typically combined: mvn clean install cleans and then runs the full default lifecycle up to install.
Build Automation Benefit: Running mvn package automatically triggers the chain: validate → compile → test → package. This enforces a disciplined, repeatable build process — developers cannot skip compilation or tests accidentally, ensuring consistent software quality in CI/CD pipelines (e.g., Jenkins calling mvn clean install).
Q12 Develop a multi-container application architecture utilizing Docker Compose. 10 Marks
Docker Compose – Multi-Container Architecture
What is Docker Compose: Docker Compose is a tool for defining and running multi-container Docker applications using a single YAML configuration file (docker-compose.yml). It allows all services (frontend, backend, database, cache) to be started, stopped, and managed with a single command.
Sample Multi-Container docker-compose.yml: The following defines a 3-tier web application with Frontend (Nginx), Backend API (Flask/Node), and a Database (MySQL).
version: '3.8' services: frontend: image: nginx:alpine ports: - "3000:80" depends_on: - backend networks: - app-network backend: build: ./backend ports: - "5000:5000" environment: - DB_HOST=database - DB_PORT=3306 depends_on: - database networks: - app-network database: image: mysql:8.0 environment: - MYSQL_ROOT_PASSWORD=secret - MYSQL_DATABASE=appdb volumes: - db-volume:/var/lib/mysql networks: - app-network networks: app-network: driver: bridge volumes: db-volume:
Services Definition: Each service block defines a container — specifying its image or build context, port mappings, environment variables, and startup dependencies (depends_on). Docker Compose ensures the database starts before the backend, and the backend before the frontend.
Networking: Docker Compose automatically creates a bridge network for all services. Containers communicate using their service names as DNS hostnames (e.g., the backend connects to the database using hostname database), eliminating the need to hardcode IP addresses.
Volumes for Persistence: Named volumes (e.g., db-volume) persist database data across container restarts. Without volumes, all database data would be lost when the container is stopped or removed.
Environment Variables: Sensitive configuration (passwords, hostnames, ports) is passed to containers via the environment block or a .env file, keeping credentials out of the application code and enabling environment-specific configurations.
Essential Docker Compose Commands:
docker compose up -d # start all services in background docker compose down # stop and remove containers/networks docker compose ps # list running services docker compose logs -f # follow logs of all services docker compose build # rebuild images docker compose scale backend=3 # scale backend to 3 instances
Advantages over Manual Docker Commands: Without Compose, each container would need to be started individually with lengthy docker run commands, and networking/volume configuration would be done manually. Compose encodes the entire application topology in one declarative file, enabling reproducibility across development, staging, and CI environments.
Use in CI/CD Pipelines: In Jenkins or GitHub Actions, docker compose up is used to spin up all required services for integration testing (test database, mock APIs), run the tests, and then docker compose down to clean up — enabling clean, isolated test environments per build run.
Q13 Evaluate the integration of Docker Trusted Registry within an enterprise deployment pipeline. 10 Marks
DTR Integration in Enterprise CI/CD Pipeline
What is Docker Trusted Registry (DTR): DTR is an enterprise-grade, private container image registry — part of Docker Enterprise. Unlike Docker Hub (public cloud), DTR is hosted on-premises or in a private cloud, giving organizations complete control over image storage, distribution, and access. It integrates with Docker Universal Control Plane (UCP) for unified management.
Role in CI Pipeline – Image Push: After a CI tool (Jenkins) builds and tags a Docker image (e.g., dtr.company.com/myapp:1.0), it pushes the image directly to DTR using docker push. DTR acts as the central artifact store for all built images, replacing Docker Hub with an internally controlled registry.
docker build -t dtr.company.com/myapp:1.0 . docker login dtr.company.com docker push dtr.company.com/myapp:1.0
Security – Role-Based Access Control (RBAC): DTR enforces granular RBAC policies — different teams (Dev, QA, Ops) get specific permissions (read-only, read-write, admin) per repository. This ensures only authorized engineers can push production images, preventing unauthorized deployments.
Security – Automated Image Scanning: DTR scans every pushed image for known CVEs (Common Vulnerabilities and Exposures) using binary-level analysis. Vulnerable images are flagged before deployment, acting as a security gate in the pipeline. Pipeline can be configured to reject images with critical vulnerabilities.
Security – Image Signing & Content Trust: DTR integrates with Docker Content Trust (DCT) to enforce that only cryptographically signed images are deployed. In the pipeline, the CI system signs the image with a private key after all tests pass; production deployments reject unsigned images.
CD Pipeline – Image Pull & Deploy: Deployment systems (Kubernetes, Docker Swarm, UCP) pull validated, signed images from DTR to staging and production environments. Since DTR is internal, this pull is fast, reliable, and not subject to Docker Hub rate limits.
High Availability & Disaster Recovery: DTR supports multi-node clustering with load balancing for high availability. Image data is replicated across nodes using shared storage. This ensures the registry remains available even if individual nodes fail — critical for uninterrupted CI/CD pipelines.
LDAP / Active Directory Integration: DTR integrates with enterprise identity providers (LDAP, Active Directory) for authentication. Engineers log in using existing corporate credentials — no separate registry accounts needed, simplifying user management and enabling SSO (Single Sign-On) for the pipeline.
DTR vs Docker Hub – Enterprise Comparison:
Feature Docker Trusted Registry (DTR) Docker Hub
Hosting Private (on-premises / private cloud) Public cloud only
Security RBAC, image signing, CVE scanning Limited (private repos only)
High Availability Clustered, load-balanced replicas No built-in HA
Authentication LDAP / Active Directory SSO Docker Hub accounts only
Compliance Regulatory compliance policies Basic compliance
Automated Garbage Collection: DTR includes scheduled garbage collection to remove unused, untagged images and reclaim storage space automatically — keeping the registry lean without manual intervention, which is essential in pipelines that generate many images per day.
